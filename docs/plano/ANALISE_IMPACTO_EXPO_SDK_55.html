1. Sumário Executivo (TL;DR)

✅ VEREDITO: VIÁVEL — atualizar a arquitetura e o plano para o Expo SDK 55 é recomendado.

Como o projeto é greenfield (zero código), isto não é uma "migração" no sentido clássico: é apenas decidir em qual SDK iniciar o scaffold. Não há retrabalho de código — o impacto concentra-se em documentação e na validação de bibliotecas contra a New Architecture (agora obrigatória).

Risco global: MÉDIO → BAIXO assim que o spike do WatermelonDB passar na Fase 0. Não há ponto eliminatório absoluto.

VereditoVIÁVEL e recomendado. Iniciar o scaffold já no SDK 55.
Risco globalMÉDIO — puxado quase inteiramente por um único item: WatermelonDB sob New Architecture obrigatória.
Ponto eliminatório?Nenhum absoluto. Há 1 ponto condicionalmente bloqueante (WatermelonDB) que já possui fallback de 1ª classe (expo-sqlite + Drizzle), isolado pela Clean Architecture.
Esforço adicional vs SDK 54Baixo a médio. O grosso é o spike de validação na Fase 0 + ajustes de app.json + Expo Router v6→v7 + nova API do expo-file-system.
Vantagem decisiva do 55A New Architecture passa de "default" a obrigatória — elimina o risco "e se a New Arch quebrar?". O ecossistema (Reanimated 4, bottom-sheet v5) já assume New Arch.
E o SDK 56?Ficar no 55 para a v1. O SDK 56 sobe o piso para iOS 16.4+, conflitando com o iOS 15+ do ADR-001 (ver §9).
A única coisa capaz de mudar este veredito é o resultado do spike do WatermelonDB na Fase 0. Se passar: segue tudo e o risco cai para BAIXO. Se falhar: aciona-se o Plano B (Drizzle) sem reescrever Domínio nem Presentation.
Resultado (Fase 0 — 19/06/2026): o spike reprovou no build — o config plugin de comunidade do WatermelonDB injeta JSIModulePackage, API removida no RN 0.83 (New Architecture). Plano B ATIVADO: o BD local é expo-sqlite + Drizzle, sem reescrever Domínio nem Presentation — exatamente como esta análise previu. Ver ADR-003 e o FASE_0_ESQUELETO_SCAFFOLD.

2. Contexto e Escopo

2.1 O que significa "migrar 54 → 55" aqui

O Sir Barbecue está em fase de planejamento: existem apenas documentos (design, arquitetura e plano de telas), nenhuma linha de código. Logo, não há upgrade incremental (53→54→55), não há prebuild a refazer e não há breaking changes a remediar em código existente. Adotar o SDK 55 = rodar o create-expo-app já no SDK 55 em vez do 54.

Consequência direta: o impacto recai sobre (1) as referências de versão e convenções nos documentos de arquitetura/plano e (2) a validação de compatibilidade das bibliotecas nativas com a New Architecture — e não sobre reescrita de telas, casos de uso ou camada de dados.

2.2 Alvo e escopo deste documento

  • Alvo da análise: Expo SDK 55 (RN 0.83.1 / React 19.2.0). Inclui uma seção dedicada ao SDK 56 (latest real), com recomendação (§9).
  • Escopo do entregável: este HTML de análise. Os documentos de arquitetura e o PLANO_IMPLEMENTACAO_TELAS.md não são alterados aqui — a §7 apenas lista, como referência, o que mudaria neles caso o veredito seja aceito.

2.3 Arquitetura definida (resumo do que está em jogo)

Clean Architecture (Presentation → Domain → Data) + offline-first; Supabase como BaaS; WatermelonDB (SQLite) como fonte primária local; Zustand para estado; Expo Router (file-based); EAS Build/Submit/Update; Sentry. O Repository Pattern isola a fonte de dados local atrás de interfaces no Domínio — fato que será determinante para a robustez do veredito.

4.4 Nota técnica decisiva — config plugin do WatermelonDB

O WatermelonDB não roda no Expo Go e precisa de config plugin para habilitar JSI no prebuild. O ecossistema de plugins está defasado em relação ao SDK 55:

PluginCobertura declaradaImplicação
@morrowdigital/watermelondb-expo-plugin (mais usado)Testado até Expo SDK 54Provavelmente funciona no 55 (wrapper fino de config nativa), mas não garantido → validar no spike.
watermelondb-expo-plugin-sdk-52-plusSDK 52/53Defasado.
skam22/watermelondb-expo-pluginaté SDK 51Defasado.
Leitura técnica: o plugin apenas injeta config nativa (pacote JSI no Android, pods no iOS) no prebuild. Como o adapter é Obj-C/Java (não C++), o risco real não é o bridgeless em si, e sim incompatibilidades de build (Gradle/AGP novos, headers do RN 0.83) que um plugin desatualizado pode não cobrir. É exatamente isso que o spike da Fase 0 valida.

6. Veredito de Viabilidade

6.1 Decisão

SIM — é viável e recomendado atualizar a arquitetura e o plano do Sir Barbecue para o Expo SDK 55. Sendo greenfield, o custo de migração de código é zero: o "upgrade" é iniciar o scaffold no 55. A maioria das libs (Reanimated 4, bottom-sheet v5, victory-native, expo-*) já assume New Architecture na linha atual — começar no 54 (New Arch opcional) seria escolher uma base que logo precisaria subir.

6.2 Classificação de risco: MÉDIO → BAIXO após o spike

FatorNívelJustificativa
WatermelonDB sob New Arch + plugin defasadoMédioAdapter Obj-C/Java (não C++) → bridgeless não é o centro do problema; o risco é build com RN 0.83/Gradle novos. Fallback existe.
Sentry Gradle 9 (Android EAS)BaixoCausa e correção conhecidas (≥ 7.3.0).
Expo Router v6→v7BaixoFile-based mantido; mudanças aditivas.
expo-file-system API novaBaixoSuperfície pequena (download de relatório).
Edge-to-edge + remoções de app.jsonBaixoAjustes mecânicos.
Demais libsBaixoNew-Arch-ready.

6.3 Ponto(s) eliminatório(s)

  • Eliminatório absoluto: NENHUM.
  • Condicionalmente bloqueante (1): WatermelonDB no spike da Fase 0 — único item capaz de travar o caminho feliz. Mitigações:
    1. Forte indício de viabilidade (adapter não-C++ roda bridgeless; há templates de mercado usando WDB em SDK 55/RN 0.83).
    2. Plano B de 1ª classe (expo-sqlite + Drizzle, melhorado no SDK 55).
    3. Clean Architecture / Repository Pattern isola a troca — Presentation e Domain não mudam.
    Mesmo no pior cenário, o projeto não fica bloqueado.

7. Plano de Execução da Adoção do SDK 55

Fase 0 — Spike de validação + tooling (a parte que decide tudo)

0.1 — Scaffold no SDK 55

  • npx create-expo-app@latest (template TS) → confirmar expo em ^55.
  • Estrutura de pastas: manter app/ (rotas) + src/ conforme o plano já documentado.
  • npx expo-doctor → tem que passar limpo.

0.2 — app.json / config (mudanças obrigatórias 54→55)

  • Remover: newArchEnabled, sdkVersion, statusBar, notification, edgeToEdgeEnabled.
  • Adicionar em plugins: config plugin do expo-notifications (substitui o campo notification), config plugin do WatermelonDB e demais plugins nativos.
  • Tratar edge-to-edge (Android) via SafeAreaProvider/insets; expo-status-bar por código.

0.3 — SPIKE WatermelonDB (gate de decisão)

  • Instalar @nozbe/watermelondb (0.28) + config plugin (@morrowdigital/watermelondb-expo-plugin como candidato primário).
  • npx expo prebuild → npx expo run:android e npx expo run:ios em device físico (não só simulador).
  • Critérios de aceite: (1) app sobe em New Architecture; (2) JSI inicializa (sem crash do adapter); (3) CRUD + transação ACID + observables reativos OK; (4) sync mínimo contra Supabase (pull/push) sem travar UI; (5) estável em iOS e Android.
  • Todos passam → segue com WatermelonDB (risco → BAIXO). Qualquer um falha → aciona o Plano B ainda na Fase 0.

0.4 — Libs restantes (sempre via expo install)

  • react-native-reanimated@~4 + react-native-worklets (+ plugin Babel); react-native-gesture-handler; @gorhom/bottom-sheet@^5.
  • react-native-gifted-charts (+ react-native-svg) primário; victory-native + Skia só se necessário.
  • react-native-webview, @react-native-community/netinfo.
  • expo-secure-store, expo-notifications, expo-auth-session, expo-web-browser, expo-file-system, expo-sharing, expo-font + @expo-google-fonts/inter.
  • @supabase/supabase-js.
  • @sentry/react-native@^7.3.0 (de pref. 7.10+) — evita o bug Gradle 9. Não usar sentry-expo.

0.5 — Disciplina nova do RN 0.83

  • Unhandled promise rejections agora são erro → padronizar try/catch em todo await (sync engine, repositórios, Supabase, file-system). Regra de lint se possível.

Fase 1+ — Ajustes pontuais (encaixam nas fases já planejadas)

  • Expo Router v6 → v7 (Fase 1): file-based mantido. Avaliar Native Tabs API p/ a bottom nav e Colors API (Material 3) p/ reforçar acessibilidade no Android. Conferir assinaturas de Stack/Tabs.
  • expo-file-system nova API (Fase 5): download/abertura de PDF/HTML com File/Directory + expo-sharing.
  • Push (Fase 5): configurar via plugin; testar em dev build (não Expo Go).
  • OAuth Google (Fase 2): considerar o universal-link callback do novo expo-web-browser.

CI/CD (EAS) — ajustes do 55

  • EAS Build: imagem com Xcode 26 (default 26.2); Node 20+ (alinhar package.json/perfil EAS).
  • EAS Update: comandos exigem --environment. Os 3 canais (development/staging/production) permanecem; só muda a flag. Atualizar eas.json/scripts.
  • Hermes bytecode diffing: opt-in no 55 → habilitar (OTA ~75% menores).
  • Sentry: garantir @sentry/react-native ≥ 7.3.0 no perfil de build (Gradle 9).

Plano B — fallback expo-sqlite + Drizzle (só se o spike 0.3 reprovar)

  1. Trocar a implementação, não as interfaces. repositories/ passam a falar com expo-sqlite + Drizzle (drizzle-orm/expo-sqlite). As interfaces de repositório no Domain ficam idênticas → Presentation/Use Cases intactos.
  2. Reatividade: usar useLiveQuery do Drizzle (observa mudanças e re-renderiza) — substitui os observables do WatermelonDB sem polling (mantém o padrão Observer).
  3. Migrations: drizzle-kit + migrate() no onInit do SQLiteProvider; Babel inline-import plugin conforme docs.
  4. Sync próprio: syncEngine/offlineQueue já são código próprio no plano — manter client_id (idempotência), synced_at/needs_sync, retry/backoff e estratégias client-wins (vendas) / server-wins (catálogo). Com Drizzle, implementa-se explicitamente o que no WDB vinha do protocolo embutido (esforço previsível, alinhado ao doc 02b).
  5. Bônus SDK 55: SQLite Inspector + tagged template literals facilitam debug/queries.
Resultado do Plano B: nenhuma tela e nenhum use case muda. Só src/data/local/ e as implementações em src/data/repositories/.
