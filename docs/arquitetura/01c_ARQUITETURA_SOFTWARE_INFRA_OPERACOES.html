8. Autenticação e Autorização

8.1 Estratégia de Autenticação

Mecanismo: Supabase Auth — JWT + OAuth 2.0 (Google)

Fluxo Email/Senha

1. POST /auth/v1/token (email + password)
2. Supabase: valida → gera access_token (JWT, 1h) + refresh_token (30d)
3. App: armazena via expo-secure-store → Keychain (iOS) / Keystore (Android)
4. Requests: Authorization: Bearer <access_token>
5. SDK renova automaticamente via refresh_token antes de expirar

Fluxo Google OAuth

1. App: abre Google OAuth via Expo AuthSession (PKCE flow)
2. Google: retorna authorization code
3. App: envia code para Supabase Auth
4. Supabase: valida com Google → cria/recupera usuário → retorna JWT
5. Na primeira vez: conta criada automaticamente (RF-02)

Estrutura do JWT

{
  "sub": "user-uuid",
  "email": "proprietaria@email.com",
  "role": "authenticated",
  "iat": 1749340800,
  "exp": 1749344400
}

8.2 Row Level Security (Autorização) — multi-tenant

Atualizado 23/06/2026: isolamento por empresa (tenant_id) via tenant_members (papéis owner/manager/employee); user_id vira auditoria. Schema canônico: SUPABASE_SCHEMA_SAAS_MULTI_TENANT.sql.

-- Todas as tabelas de negócio têm RLS + coluna tenant_id
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

-- Usuário só acessa dados das empresas a que pertence
CREATE POLICY "tenant_all" ON products
  USING (tenant_id IN (SELECT public.user_tenant_ids()))
  WITH CHECK (tenant_id IN (SELECT public.user_tenant_ids()));
-- Custom Access Token Hook injeta tenant_ids no JWT (caminho rápido).

CamadaTecnologiaTTLO que cachear
Local SQLiteWatermelonDBPermanente (até sync)Todos os dados operacionais

Dashboard dataWatermelonDB + timestamp5 minutos (RF-28)Aggregates do dashboard
RelatóriosSupabase StoragePermanentePDFs e HTMLs já gerados

Estratégia por Tipo de Dado

  • Catálogo (produtos, categorias, fornecedores): Write-through — escrita simultânea local e remota quando online
  • Vendas: Write-local-first — gravada localmente antes de qualquer confirmação; sync em background
  • Dashboard: Cache-Aside com TTL 5 min; exibe timestamp da última atualização
  • Relatórios gerados: URL armazenada localmente — não regenera arquivo existente

14. Escalabilidade e Resiliência

Referência DDIA: Cap. 1 e Cap. 8 — problemas em sistemas distribuídos.

14.1 Resiliência no App Mobile

CenárioComportamentoImplementação
Supabase indisponívelOpera 100% offlineWatermelonDB como fonte primária
Falha de syncDados não perdidosneeds_sync=true nunca descartado sem confirmação
Crash durante vendaTransação local garante consistênciaWatermelonDB — transações ACID no SQLite
Troca de dispositivoDados sincronizam no novo deviceRe-sync completo ao fazer login
Relatório falhaStatus failed + notificaçãoUsuário pode solicitar novamente

14.2 Garantias de Durabilidade

  • Vendas: gravadas no SQLite local antes de confirmar sucesso ao usuário
  • Sync: retry exponencial; dados ficam na fila indefinidamente até sync bem-sucedido
  • Backups: daily automático no Supabase (7 dias no free tier)
  • Retenção 5 anos: PostgreSQL mantém dados de vendas por 5 anos (RNF-11 — obrigação fiscal)

16. Riscos e Trade-offs

RiscoProbabilidadeImpactoMitigação
Vendor lock-in no SupabaseMédiaMédioPostgreSQL padrão — export via pg_dump; Supabase é open-source
Limite free Supabase (500MB)BaixaMédioMonitorar; upgrade Pro ($25/mês) antes de 80%
WatermelonDB sob New Arch (SDK 55) — resolvido——Gate reprovou no build (JSIModulePackage removida no RN 0.83) → Plano B ATIVADO: expo-sqlite (first-party) + Drizzle, sem plugin de comunidade. Ver ADR-003.
Edge Function timeout em relatórioBaixaMédioTimeout 150s; relatório gerencial estimado < 30s; margem suficiente
Conflitos de sync longos offlineBaixaMédioEstratégias de resolução definidas por entidade; vendas nunca perdem dados
Acessibilidade insuficiente para deficiência visualMédiaAltoTestes com VoiceOver/TalkBack desde o desenvolvimento; revisão por especialista em acessibilidade
App rejeitado na App StoreBaixaAltoSeguir guidelines Apple; declarar usos corretos de permissões; LGPD compliance
Referência DDIA: Cap. 12 — Kleppmann discute os trade-offs fundamentais. Para Sir Barbecue, o trade-off central é simplicidade operacional (BaaS) vs controle total. Supabase é correto para o tamanho atual — revisável na v2 se volume ou complexidade crescerem.
