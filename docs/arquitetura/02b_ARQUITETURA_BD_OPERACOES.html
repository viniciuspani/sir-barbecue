5. Estratégia de Índices

Referência DDIA: Cap. 3 — B-Trees vs LSM-Trees e como índices afetam performance de leitura e escrita.

Índices: tabela sales

-- Índice principal: consultas de relatório (user + data)
CREATE INDEX idx_sales_user_date ON sales(user_id, sale_date DESC)
WHERE deleted_at IS NULL;

-- Por forma de pagamento (relatório diário RF-17)
CREATE INDEX idx_sales_user_payment ON sales(user_id, payment_method, sale_date);

-- Vendas não sincronizadas (sync offline)
CREATE INDEX idx_sales_not_synced ON sales(user_id, synced_at)
WHERE synced_at IS NULL;

Índices: tabela sale_items

CREATE INDEX idx_sale_items_sale    ON sale_items(sale_id);
CREATE INDEX idx_sale_items_product ON sale_items(product_id);

Índices: tabela stock_items

CREATE INDEX idx_stock_items_user  ON stock_items(user_id);
CREATE INDEX idx_stock_items_alert ON stock_items(user_id, quantity, alert_threshold)
WHERE alert_threshold > 0;

Índices: tabela products

CREATE INDEX idx_products_user_active ON products(user_id, is_active, category_id)
WHERE is_active = true;

6. Transações e Controle de Concorrência

Referência DDIA: Cap. 7 — Kleppmann desmistifica ACID, níveis de isolamento e write skew.

Nível de Isolamento: READ COMMITTED

Padrão do PostgreSQL. Suficiente para 1 usuário com baixíssimo volume. Idempotência por client_id cobre os casos de retry.

Fluxo: Registro de Venda (RF-12 + RF-10)

BEGIN;

-- 1. Inserir venda (idempotente)
INSERT INTO sales (user_id, total_amount, payment_method, consumption_mode, sale_date, client_id)
VALUES ($1, $2, $3, $4, $5, $6)
ON CONFLICT (client_id) DO NOTHING
RETURNING id;

-- 2. Inserir itens — trigger deduct_stock_on_sale executa automaticamente
INSERT INTO sale_items (sale_id, product_id, quantity, unit_price)
VALUES ($sale_id, $pid1, $qty1, $price1), ($sale_id, $pid2, $qty2, $price2);

COMMIT;
-- Garantia: venda + itens + dedução de estoque ocorrem atomicamente

Proteções contra Race Conditions

CenárioProteção
Mesma venda enviada duas vezes (retry)UNIQUE (client_id) + ON CONFLICT DO NOTHING
Saldo negativo de estoqueCHECK (quantity >= 0) em stock_items
Produto duplicado por nomeUNIQUE (user_id, name) em products

Boas Práticas

  • Sempre fazer: migrations aditivas (ADD COLUMN, ADD TABLE, ADD INDEX)
  • Sempre fazer: ADD COLUMN ... DEFAULT value em PostgreSQL 11+ é O(1)
  • Evitar: DROP COLUMN sem período de deprecação (mínimo 1 sprint)
  • Evitar: renomear colunas diretamente — adicionar nova → migrar → remover antiga após 1 deploy

11. Queries Críticas

Referência DDIA: Cap. 3 — B-Tree indexes e quando cada abordagem é eficiente.

Relatório Diário de Vendas (RF-17)

SELECT
    payment_method,
    COUNT(*)          AS total_transactions,
    SUM(total_amount) AS total_revenue
FROM sales
WHERE
    user_id = $1
    AND sale_date >= $date_start
    AND sale_date <  $date_end
GROUP BY payment_method
ORDER BY payment_method;
-- Usa: idx_sales_user_date → Index Scan

Relatório de Produtos com Margem (RF-19, RF-20)

SELECT
    p.name                                           AS product_name,
    SUM(si.quantity)                                 AS total_quantity,
    SUM(si.subtotal)                                 AS total_revenue,
    COALESCE(ps.purchase_price, 0)                   AS unit_cost,
    SUM(si.subtotal) -
      (COALESCE(ps.purchase_price, 0) * SUM(si.quantity)) AS gross_profit,
    ROUND(
      ((SUM(si.subtotal) - COALESCE(ps.purchase_price, 0) * SUM(si.quantity))
       / NULLIF(SUM(si.subtotal), 0)) * 100, 2
    ) AS margin_pct
FROM sale_items si
JOIN sales s   ON s.id = si.sale_id
JOIN products p ON p.id = si.product_id
LEFT JOIN product_suppliers ps
    ON ps.product_id = p.id AND ps.is_preferred = true
WHERE s.user_id = $1
  AND s.sale_date >= $date_start
  AND s.sale_date <  $date_end
GROUP BY p.id, p.name, ps.purchase_price
ORDER BY total_revenue DESC;

Paginação por Cursor (histórico de vendas)

-- ❌ Ruim (offset): performance degrada em páginas altas
SELECT * FROM sales WHERE user_id = $1 ORDER BY sale_date DESC LIMIT 20 OFFSET 10000;

-- ✅ Bom (cursor): performance constante
SELECT * FROM sales
WHERE user_id = $1
  AND sale_date < $cursor_date  -- timestamp da última venda da página anterior
ORDER BY sale_date DESC
LIMIT 20;

DELETE FROM stock_entries WHERE user_id = $user_id;
DELETE FROM sales         WHERE user_id = $user_id; -- CASCADE → sale_items

SELECT auth.delete_user($user_id); -- Remove de auth.users
Atenção: para compliance fiscal (RNF-11), os dados de sales e sale_items devem ser retidos por 5 anos. Em exclusão de conta, avaliar anonimização dos registros (NULL no user_id) ao invés de deleção hard.

Queries de Monitoramento

-- Top queries mais lentas
SELECT LEFT(query, 100) AS query_preview, calls,
       ROUND(mean_exec_time::numeric, 2) AS avg_ms
FROM pg_stat_statements
ORDER BY mean_exec_time DESC LIMIT 10;

-- Tamanho das tabelas
SELECT relname AS table_name,
       pg_size_pretty(pg_total_relation_size(oid)) AS total_size
FROM pg_class
WHERE relkind = 'r' AND relnamespace = 'public'::regnamespace
ORDER BY pg_total_relation_size(oid) DESC;

-- Alertas de estoque ativo (RF-11)
SELECT p.name, si.quantity, si.alert_threshold
FROM stock_items si
JOIN products p ON p.id = si.product_id
WHERE si.quantity <= si.alert_threshold
  AND si.alert_threshold > 0
  AND p.is_active = true;
