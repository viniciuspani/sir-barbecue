5.3 Fluxo — Registro de Venda (Offline-First)

sequenceDiagram
    actor Owner as Proprietária
    participant App as App (UI + Domain)
    participant Local as WatermelonDB (Local)
    participant Queue as Offline Queue
    participant Sync as Sync Engine
    participant Remote as Supabase PostgreSQL

    Owner->>App: Seleciona produtos + quantidade + pagamento
    App->>App: Valida disponibilidade em estoque local
    App->>Local: Salva venda + itens (INSERT)
    Local-->>App: Confirmado
    App->>Local: Decrementa estoque (UPDATE)
    App-->>Owner: "Venda registrada ✓"

    alt Online
        Sync->>Remote: Sync venda para servidor
        Remote-->>Sync: Confirmado
    else Offline
        App->>Queue: Adiciona venda à fila de sync
        Queue-->>Owner: Indicação visual "modo offline"
    end

    Note over Sync: Ao reconectar: processa fila
    Sync->>Remote: Envia vendas pendentes
    Remote-->>Sync: ACK
    Sync-->>Owner: "Sincronização concluída ✓"

6. Design de APIs e Contratos

Referência DDIA: Cap. 4 — evolução de APIs sem quebrar compatibilidade. Supabase expõe REST API auto-gerada via PostgREST, complementada por Edge Functions para lógica complexa.

6.1 Estratégia de API

  • CRUD simples → Supabase REST API (auto-gerada via PostgREST)
  • Lógica complexa (geração de relatório, cálculo de margens) → Edge Functions
  • Realtime (sync, dashboard) → Supabase Realtime WebSocket
  • Base URL: https://<project>.supabase.co/rest/v1/
  • Autenticação: Authorization: Bearer <jwt_token> em todos os requests
  • Row Level Security (RLS) garante isolamento por usuário

7.3 Fluxo de Sincronização Offline

OFFLINE:
  Operação → WatermelonDB local (write) → marcado como "needs_sync=true"

RECONEXÃO DETECTADA:
  Sync Engine lê registros com needs_sync=true
  → POST/PATCH no Supabase para cada registro
  → 200 OK: marca needs_sync=false
  → Conflito: "client wins" para vendas; "server wins" para catálogo
  → Zustand: sync_status = 'synced'
  → UI: confirmação visual

ONLINE CONTÍNUO:
  Realtime WebSocket → mudanças do servidor → WatermelonDB local
  → UI atualiza via WatermelonDB observables

7.4 Estratégia de Retry e Idempotência

Tentativas: 3 (backoff exponencial: 1s → 2s → 4s)
Falha definitiva: operação permanece na fila para próxima sync

Idempotência: todas as operações incluem client_id (UUID gerado offline)
  → Servidor: ON CONFLICT (client_id) DO NOTHING
