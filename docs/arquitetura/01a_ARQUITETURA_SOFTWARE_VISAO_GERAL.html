1. Visão Geral do Sistema

1.1 Propósito

Sir Barbecue é um aplicativo mobile para gestão completa de um trailer de churrasquinho. Resolve o problema de controle operacional informal (cadernos, planilhas, memória) ao centralizar no smartphone do proprietário: registro de vendas, controle de estoque, gestão de fornecedores e geração de relatórios financeiros — com suporte completo a operação sem internet.

Usuário-alvo: Proprietária do negócio, perfil não técnico. Uma das usuárias possui deficiência visual parcial (enxerga apenas com um olho), o que impõe requisitos específicos de acessibilidade e legibilidade.

1.2 Características Críticas do Sistema

2. Decisões Arquiteturais (ADR)

ADR = Architecture Decision Record. Cada decisão relevante documentada com alternativas e motivo da escolha.

ADR-001 — Framework Mobile: React Native (Expo) vs Flutter

StatusAceito
Data07/06/2026

Contexto

O sistema deve funcionar em iOS 15+ e Android 10+ com uma única base de código, com bom suporte a SQLite local para offline e integração nativa com recursos de acessibilidade (VoiceOver/TalkBack).

Alternativas Consideradas

AlternativaPrósContras
React Native + ExpoEcossistema JS/TS maduro; Expo simplifica build; integração nativa com Supabase/Firebase; WatermelonDB para offline; APIs de acessibilidade nativasPerformance ligeiramente inferior ao Flutter em animações pesadas
FlutterPerformance nativa excelente; Dart forteEcossistema menor; integração Supabase menos madura; Dart menos difundido
Nativo (SwiftUI + Compose)Performance máximaDois codebases, custo dobrado
Decisão: React Native com Expo (Managed Workflow). Ecossistema TypeScript, integração direta com Supabase SDK, WatermelonDB para offline-first e APIs de acessibilidade nativas são os fatores determinantes.

Atualização SDK 55 (19/06/2026): alvo passa a Expo SDK 55 / RN 0.83.1 / React 19.2; a New Architecture deixa de ser opcional e torna-se obrigatória (Legacy removida). Piso de iOS 15+ mantido (o SDK 56 exigiria iOS 16.4+). Ver análise de impacto.

ADR-002 — Backend: Supabase (BaaS) vs Backend Customizado

StatusAceito
Data07/06/2026

Contexto

O sistema tem um único usuário com baixo volume de dados. Montar infraestrutura customizada representa overhead desproporcional ao tamanho do negócio.

Alternativas Consideradas

AlternativaPrósContras
SupabasePostgreSQL gerenciado; Auth nativa (email + OAuth); Storage; Edge Functions; Realtime; plano gratuito generosoVendor lock-in; limites no plano free
FirebaseMaduro; offline excelente com FirestoreNoSQL — menos adequado para relatórios com JOINs; custo pode escalar
Node.js + PostgreSQL customizadoControle total; sem lock-inMuito overhead para 1 usuário; CI/CD, Docker, infra necessários
Decisão: Supabase. PostgreSQL ideal para relatórios com JOINs complexos. Auth built-in com Google OAuth. Elimina servidor, Docker e CI/CD em v1. Mitigação de lock-in: banco é PostgreSQL padrão — pg_dump disponível.

ADR-003 — Estratégia Offline: WatermelonDB vs SQLite Manual

StatusAceito
Data07/06/2026

Contexto

RF-15 e RF-16 exigem registro de vendas sem internet e sincronização automática ao reconectar. Esta é a funcionalidade mais crítica — a proprietária opera em locais com sinal instável.

Alternativas Consideradas

AlternativaPrósContras
WatermelonDBProjetado para React Native; SQLite local; API de sync built-in; performance excelente; trata conflitosComplexidade inicial de setup; curva de aprendizado
Expo SQLite + sync manualControle totalMuito trabalho manual; difícil tratar conflitos
RealmBom para mobileSync requer MongoDB Atlas; custo adicional
Decisão: WatermelonDB com Supabase como servidor de sync. Todas as operações operam sobre o banco local — Supabase é tratado como destino de sincronização. Conflitos em vendas offline: estratégia "client wins" com timestamp.

Não há release estável do plugin p/ SDK 55 (só 2.4.0-beta); o suporte New Arch do WDB tem issues abertas (#1769/#1851).

ADR-004 — Geração de Relatórios: Edge Functions vs No Dispositivo

StatusAceito
Data07/06/2026

Contexto

RF-21 a RF-26 requerem geração assíncrona de relatórios em PDF e HTML com gráficos, sem bloquear o app (RNF-02).

Decisão: Supabase Edge Functions para geração server-side. Usuário solicita → Edge Function processa em background → push notification avisa quando pronto → PDF armazenado no Supabase Storage para download.

ADR-005 — Autenticação: Supabase Auth

Decisão: Supabase Auth (incluso). Suporta email/senha com verificação e Google OAuth nativo. Tokens JWT com refresh automático armazenados no Keychain (iOS) e Keystore (Android) via SDK.

ADR-006 — Estado Mobile: Zustand

Decisão: Zustand para estado global do app (auth state, sync status, connectivity). WatermelonDB gerencia estado de dados via observables reativos. Redux seria overkill para app de usuário único.

3. Estilo e Padrão Arquitetural

3.1 Estilo Arquitetural Adotado

Estilo Mobile: Clean Architecture (Presentation → Domain → Data)

Estilo Backend: Backend as a Service (BaaS) via Supabase — sem servidor customizado em v1

Padrão principal: Offline-First — o app funciona integralmente sem internet; o servidor é tratado como mecanismo de sincronização e persistência durável, não como dependência de operação.
