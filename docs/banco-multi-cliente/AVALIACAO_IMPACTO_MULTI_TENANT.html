1. Sumário Executivo (veredito)

✅ VIÁVEL — a migração para multi-tenant é incremental e bem isolada.

O isolamento de dados passa de user_id (o dono) para tenant_id (a empresa). O impacto no app já construído é concentrado em 3 pontos de código (authStore, syncEngine, signup) + setup do Supabase. A Clean Architecture isola de novo: domínio, telas, repositórios locais e schema local ficam praticamente intactos.

Risco global: MÉDIO — sobretudo pela segurança 100% dependente da RLS (sem isolamento físico entre empresas) e pela necessidade de habilitar o Custom Access Token Hook.

VereditoVIÁVEL e recomendado. Migração incremental, sem reescrita do app.
Risco globalMÉDIO — segurança via RLS, hook obrigatório, e testes de isolamento (tenant X ≠ tenant Y).
Esforço de códigoPequeno-médio: 3 arquivos (authStore, syncEngine, signup) + leitura de claim. Domínio/telas/repos locais inalterados.
Decisão MVP1 empresa por usuário (o schema suporta multi-empresa + equipe; seletor e convites ficam para fase posterior).
Pré-requisito de runtimeRodar o SQL multi-tenant + habilitar o Custom Access Token Hook no Dashboard do Supabase.
Dados existentesComo ainda é fase de teste → recomenda-se banco fresh (evita backfill de tenant_id).

3. O modelo multi-tenant (como funciona)

3.1 Empresa e equipe

  • tenants — a empresa/estabelecimento (nome fantasia, CNPJ, logo, owner_user_id). É a unidade de isolamento.
  • tenant_members — liga user_id ↔ tenant_id com role (owner/manager/employee). Permite vários usuários por empresa.

3.2 RLS sem recursão

Funções SECURITY DEFINER que leem tenant_members ignorando a RLS (evita recursão):

  • user_tenant_ids() → empresas do usuário logado. Base de toda a RLS: tenant_id in (select user_tenant_ids()).
  • is_tenant_owner(tenant_id) → se o usuário é owner (gestão de equipe e dados cadastrais da empresa).

3.3 Custom Access Token Hook (otimização) — tipo Postgres

add_tenant_claims é uma função Postgres (PL/pgSQL) — por isso, no Dashboard, o hook é do tipo Postgres (schema public, função add_tenant_claims), não HTTPS (HTTPS seria para chamar uma URL externa). Ela injeta app_metadata.tenant_ids no JWT a cada login.

Após habilitar o hook, faça logout/login num usuário de teste: tokens antigos não carregam o claim até renovar (~1h) ou novo login.

Dois caminhos para a RLS descobrir as empresas do usuário

O resultado é o mesmo; muda a fonte dos tenant_ids:

Caminho padrão (aplicado pelo script)Caminho rápido (comentado no script — linhas 426–429)
Expressão na policytenant_id in (select public.user_tenant_ids())tenant_id = any (select (jsonb_array_elements_text(auth.jwt() -> 'app_metadata' -> 'tenant_ids'))::uuid)
Fonte dos tenant_idsSELECT em tenant_members a cada query (função SECURITY DEFINER)Direto do JWT (injetado pelo hook) — sem tocar no banco
CustoUma consulta extra por query~Zero (já está no token)
Atualização (frescor)Sempre atual — mudou a equipe, vale na horaDefasado até o token renovar (~1h) ou novo login
Trade-off: o caminho rápido troca frescor por custo/latência. Recomendação: no MVP (1 empresa/usuário), ficar no caminho padrão (user_tenant_ids()) — sempre correto e com custo irrelevante no volume atual. Migrar para o JWT só se custo/latência pesarem com muitos clientes; nesse caso, trocar (select public.user_tenant_ids()) pela expressão do JWT em todas as policies (inclusive dentro do EXISTS das tabelas filhas, onde vira p.tenant_id = any (...)). Dica: encapsular a expressão num helper auth_tenant_ids() para manter DRY e alternar fácil entre os dois caminhos.

3.4 Bootstrap e seed

  • handle_new_user (trigger em auth.users) → ao criar usuário, cria a 1ª empresa (nome = raw_user_meta_data.business_name ou "Minha Empresa") + tenant_members(owner).
  • seed_tenant_categories (trigger em tenants) → ao criar empresa, semeia 4 categorias padrão.

5. Detalhe das mudanças de código

5.1 authStore — carregar o tenant atual

Após o login, derivar o tenant do claim do JWT (ou, como fallback, consultar tenant_members):

// após getSession()/onAuthStateChange:
const tenantIds = (session?.user?.app_metadata?.tenant_ids ?? []) as string[];
let currentTenantId = tenantIds[0] ?? null;
if (!currentTenantId) {
  // fallback se o hook não estiver habilitado
  const { data } = await supabase.from('tenant_members')
    .select('tenant_id').limit(1).maybeSingle();
  currentTenantId = data?.tenant_id ?? null;
}
set({ session, user: session?.user ?? null, currentTenantId });

Expor currentTenantId no estado para o Sync Engine consumir. (MVP: 1 empresa → pega a 1ª.)

5.2 syncEngine.ts — enviar tenant_id no push

  • Tabelas com tenant_id direto (products, sales, categories, suppliers, stock_items, stock_entries, reports): o payload do upsert passa a incluir tenant_id: currentTenantId.
  • Filhas NÃO mudam (sale_items, product_suppliers, product_day_visibility) — herdam o tenant do pai; a RLS valida via EXISTS.
  • Pull: filtrar pela empresa atual — .eq('tenant_id', currentTenantId) — para não misturar empresas caso o usuário pertença a várias.
  • runSync() passa a exigir currentTenantId (no-op sem tenant, como já é no-op sem sessão).
// pushProducts — exemplo
upsertRemote('products', rows.map((r) => ({
  client_id: r.id,
  tenant_id: currentTenantId,        // ← novo
  name: r.name, price: r.price,
  is_active: r.isActive,
  category_client_id: r.categoryId,
})));

5.3 signup — nome da empresa

  • Adicionar campo "Nome da empresa" na tela de cadastro.
  • signUpWithEmail(email, password, businessName) → supabase.auth.signUp({ email, password, options: { data: { business_name: businessName } } }). O trigger handle_new_user nomeia a 1ª empresa com esse valor.
  • Google OAuth: não há como coletar o nome antes → a empresa nasce como "Minha Empresa" (renomeável na Fase 5 — "Minha Empresa").

5.4 supabaseClient

Sem mudança estrutural. O flowType: 'pkce' e o storage (SecureStore) permanecem. O claim tenant_ids chega no JWT quando o hook está habilitado.

6. Setup do Supabase (pré-requisito do runtime)

  1. Rodar o SUPABASE_SCHEMA_SAAS_MULTI_TENANT.sql no SQL Editor (substitui o SUPABASE_SCHEMA_COMPLETO.sql single-user; idempotente). Se o banco já tinha o single-user com dados, preferir banco fresh.
  2. Habilitar o Custom Access Token Hook (tipo Postgres): Dashboard → Authentication → Hooks → Custom Access Token → schema public, função add_tenant_claims. Depois, logout/login para o novo JWT trazer o claim. Sem o hook, o app usa o fallback por query (mais lento, mas funciona — ver §3.3 para os dois caminhos da RLS).
  3. No cadastro, enviar o metadado business_name (já tratado pela mudança no signup).
  4. Testar isolamento: usuário do tenant X não deve ler/escrever linhas do tenant Y.
O SUPABASE_SCHEMA_COMPLETO.sql (single-user) fica obsoleto com a adoção do multi-tenant.

7. Impacto nas Fases 4+ (a fazer)

  • Fase 4 (Produtos/Vendas/Estoque/Dashboard): as telas continuam iguais; o tenant_id é carregado pela camada de sync (não pela UI). Queries locais já são da empresa (device = 1 empresa).
  • Fase 5 (Config): ganha a tela "Minha Empresa" — renomear empresa/CNPJ/logo (owner) e, opcionalmente, gerenciar equipe (convidar/remover membros = inserir/remover em tenant_members, owner-only). É um RF novo (multi-tenant).
  • Seletor de empresa: só se o produto evoluir para um usuário em várias empresas. Fora do MVP.
  • Exclusão de conta (RNF-08): passa a considerar a empresa (excluir empresa vs sair da empresa) — refinar na Edge Function.

8. Riscos e decisões

ItemNívelObservação / mitigação
Segurança 100% na RLS (sem isolamento físico)MédioUma empresa nunca pode ver outra. Testar todas as policies com 2 tenants antes de clientes reais.
Custom Access Token Hook não habilitadoBaixoO claim não chega → usar o fallback por query em tenant_members (já previsto no authStore).
Banco local single-tenantBaixoMVP = 1 empresa. Se um dia houver troca de empresa no mesmo device, resetar o DB local (ou escopar por tenant).
Migração de dados existentesBaixoAinda é teste → banco fresh. Se houvesse dados: criar tenant + backfill tenant_id a partir do user_id.
Free tier (multi-cliente real)Médio500MB DB / 1GB Storage; projeto pausa após ~7 dias inativo. Para clientes reais, avaliar plano Pro.
Google OAuth sem nome de empresaBaixoEmpresa nasce "Minha Empresa" → renomear em "Minha Empresa" (Fase 5).

Decisões adotadas

  • MVP = 1 empresa por usuário (estrutura pronta para multi-empresa + equipe; seletor/convites depois).
  • Filhas normalizadas (sem tenant_id próprio) — a desnormalização (Sugestão 4) só se a RLS via EXISTS virar gargalo com volume alto.

9. Veredito final + checklist de migração

SIM — migrar para multi-tenant é viável e de baixo atrito no app. A mudança é majoritariamente de schema/RLS no Supabase + 3 ajustes de código (authStore, syncEngine, signup). A Clean Architecture mantém domínio, telas e repositórios locais intactos.

Checklist da migração (código — etapa seguinte, sob novo OK)

  • ☐ authStore: carregar currentTenantId (claim do JWT / fallback query).
  • ☐ syncEngine: enviar tenant_id no push das tabelas diretas; filtrar pull por tenant; exigir tenant no runSync.
  • ☐ signup + auth.ts: campo "Nome da empresa" → business_name.
  • ☐ Gate local: tsc + eslint + expo export verdes.

Checklist do backend (você)

  • ☐ Rodar o SUPABASE_SCHEMA_SAAS_MULTI_TENANT.sql (banco fresh).
  • ☐ Habilitar o Custom Access Token Hook (add_tenant_claims).
  • ☐ Testar isolamento entre 2 empresas.

Docs a atualizar

  • ☐ 01a (novo ADR de multi-tenancy / auth), 01c §8 (modelo passa de "proprietário único" → multi-tenant), 02a/02b (tenant_id + tenants/tenant_members), e o PLANO (Fases 4+ tenant-aware + "Minha Empresa"/equipe).
